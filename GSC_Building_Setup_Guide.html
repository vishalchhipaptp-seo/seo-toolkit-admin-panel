Setting Up Your Building's GSC Workbench

This gives your building its own private workspace for managing Google Search Console accounts — separate from every other building's. Logging in and who's an admin is still handled centrally by the super admin; this guide is only about GSC.

ⓘ PTP is already done — skip this whole guide if you're PTP. This is for a new building (313, 188, etc.), or an admin adding/checking GSC accounts for one that already exists.
1You'll need the file building_gsc_apps_script.js (given to you) 2You'll need the file GSC_Building_Template.xlsx (given to you)
1

Create your building's spreadsheet

This is where all of this building's GSC accounts and settings will live.

  1. In Google Drive, click New → Google Sheets → Blank. Name it something like "313 – GSC Tracker".
  2. Click File → Import → Upload, choose GSC_Building_Template.xlsx, then pick Replace spreadsheet.
  3. Copy the long ID from the sheet's URL — the part between /d/ and /edit. You'll need it in a minute.
2

Add the script

This is the code that actually runs your building's GSC connections.

  1. In your new sheet, open Extensions → Apps Script.
  2. Delete whatever code is already there, and paste in the entire contents of building_gsc_apps_script.js. Save.
  3. On the left, click the + next to Libraries, add this ID, then click Add:
    Library ID1B7FSrk5Zi6L1rSxxTDgDEUsPzlukDsi4KGuTMorsTQHhGBzBkMun4iDF
3

Enter three settings

Open Project Settings → Script Properties → Add property and add these three:

SHEET_IDthe ID you copied in Step 1 CLIENT_IDfrom Step 4 below CLIENT_SECRETfrom Step 4 below
Skip:Don't add anything called ADMIN_KEY — it isn't used anymore. Admin actions now check the admin's own SEO Toolkit Pro login instead.
4

Get your OAuth credentials

This is what lets the script securely talk to Google on your behalf.

  1. Easiest option: reuse PTP's existing Client ID / Client Secret. Enter them into the two fields from Step 3.
  2. Reload the sheet, then run GSC Tools → Show owner / project info — it prints a redirect URI ending in /usercallback.
  3. In Google Cloud Console, open that OAuth client under APIs & Services → Credentials, add that URI under Authorized redirect URIs, and save.
Heads up:If this building will connect close to 100 Google accounts, give it its own Cloud project + OAuth client instead of sharing PTP's — Google limits how many accounts one unverified client can serve.
5

Publish it as a web app

  1. In the Apps Script editor: Deploy → New deployment → Web app.
  2. Set Execute as: Me and Who has access: Anyone, then Deploy.
  3. Approve the Google permission prompt.
  4. Copy the web address it gives you (it ends in /exec).
6

Send the link to the super admin

They'll paste it into the central sheet so your building's users get routed to your GSC accounts, and no one else's.

7

Connect your Google accounts

Reload the sheet first so the GSC Tools menu shows up. Repeat these five steps for each account:

  1. Add a row to the GSC_Config tab: _setup | new@email.com | accountkeyname
  2. GSC Tools → Generate auth URLs
  3. Open the Auth_Status tab, click that account's link, and click Allow.
  4. GSC Tools → Check authorisation status — confirm it says Authorised.
  5. GSC Tools → Sync domains from GSC — pulls in every site that account can access.
An account's connection broke — how do I fix just that one?

GSC Tools → Check authorisation status → Reset ONE account, enter its key, then Generate auth URLs and repeat steps 3–4 above for that account only.

The full menu reference lives in the GSC_Help tab of your sheet.

7b

Optional: use your own API key for one service

Most buildings can skip this — you automatically inherit the shared keys.

  1. Open the Keys tab (Key Name | Value | Description).
  2. Fill in Value only for a key you specifically want to override for this building — e.g. your own screenshot-hosting key.
  3. Done — everyone in your building picks it up automatically within about 15 minutes. Nothing to restart.
8

Set your alert email

  1. Open the GSC_Settings tab.
  2. Set Alert Email TO to your building's admin email (comma-separate for more than one).
Important:Skip this and every alert — Manual Actions, Security Issues, removals, and the checks in Step 9 — silently goes to PTP's inbox instead of yours.
9

The daily version check (sets itself up)

This checks that the GSC access you have for a domain actually matches the URL version the site really runs on — e.g. you're set up for https://www. but the site now runs on plain https://. Nothing for you to configure:

  • A quick check already runs automatically every ~6 hours, for any domain whose access just changed.
  • A full sweep of every domain runs automatically once a day at 3am, and emails your Alert Email TO address if anything needs attention.
I want to run a full check right now, or something's stuck

Run it on demand any time: GSC Tools → Run full GSC version check now. You'll get an email once it finishes.

If it stops early with a permission error mentioning script.scriptapp: open the Apps Script editor, pick any function from the dropdown (e.g. dailyVersionCheck) and click Run once to re-approve that permission, then run the full check again.

If you just edited GSC_Config by hand, run clearDomainAccountMapCache once from the Apps Script editor so the change is picked up immediately instead of within 5 minutes.

What the super admin still needs to do

  • In the central sheet's Buildings tab, paste this building's /exec link into gsc_script_url.
  • Set the building's manager as its admin (role = admin, in that building's own users sheet).
  • Nothing else — shared default keys already live in the central Keys tab, and every building inherits them unless it set its own override in Step 7b.
